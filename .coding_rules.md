# RepositoryAutomation コーディング規約 (v1.0)

このドキュメントは、本リポジトリにおける自動コード生成の絶対的な基準である。AIエージェント（Llama/Qwen/Aider）は、以下の規約を遵守しなければならない。

## 1. アーキテクチャ原則
- **Vertical Slicing**: 機能は `src/features/{FeatureName}` 配下で完結させること。他機能の内部実装（Entity/UseCase）に直接依存してはならない。
- **Clean Architecture**: 以下の3層構造を厳守すること。
    - `Entity/`: ドメインモデルとビジネスロジック。外部ライブラリに依存しない。
    - `Interface/`: 外部との境界（入力ポート）。
    - `UseCase/`: アプリケーションロジック。Interfaceを実装し、Entityを操作する。
- **DIP (依存性逆転の原則)**: ユースケースは具体的なインフラ（DBやAPI）に依存せず、インターフェースを介して抽象に依存すること。

## 2. 実装ガイドライン (C# / .NET)
- **命名規則**: 
    - クラス/メソッド名: `PascalCase`
    - 変数/引数名: `camelCase`
    - インターフェース名: `I` から始まる `PascalCase`
- **名前空間**: `App.Features.{FeatureName}.{Layer}` の形式を維持すること。
- **型定義**: 
    - プリミティブ型よりもドメイン固有の型（または適切なC#標準型）を優先する。
    - Null許容参照型 (`?`) を適切に使い、実行時エラーを防止する。
- **非同期**: I/Oが想定される処理には `Task` / `async` / `await` を検討すること（要件に明示がある場合）。

## 3. 禁止事項
- **スコープ外編集**: `src/features/{FeatureName}` 以外のディレクトリにあるファイルを編集してはならない。
- **破壊的変更**: `Shared` や `Core` にある既存の共通クラスを、AIの判断だけで書き換えてはならない。変更が必要な場合は `Conflict` として報告すること。
- **ライブラリの勝手な導入**: `NuGet` パッケージを独断で追加してはならない。標準ライブラリのみで実装すること。

## 4. Gatekeeper (Reasoning) 用の判定基準
Issueの解析時、以下のいずれかに該当する場合は実装を拒否（Conflict）すること。
1. **要件不足**: メソッド名、引数、戻り値のいずれかが不明確。
2. **規律違反**: クリーンアーキテクチャの階層構造を無視するような要求。
3. **リソース過負荷**: 一度のIssueで複数の機能を同時に作成・変更しようとする要求（1 Issue = 1 Feature の原則）。

- **コメント規約**:
    - ロジックの意図を説明する日本語コメントを積極的に記述すること。
    - 公開メソッドには `///` (XMLドキュメント形式) で日本語の引数・戻り値説明を付けること。
    - ファイルは必ず **UTF-8 (BOMなし推奨)** で保存し、エンコーディングエラーを防止すること。
